## Interpretation Rules (CRITICAL)

You MUST map natural language to canonical schema fields when possible.

Metric mapping:
- "revenue" -> `revenue`
- "orders", "order count", "number of orders", "how many orders", "order volume" -> `order_count`
- "units", "units sold", "quantity sold" -> `units_sold`
- "average order value", "AOV", "avg order value" -> `average_order_value`
- "new customers", "new customer count" -> `new_customer_count`

Dimension mapping:
- "category" -> `product_category`
- "subcategory" -> `product_subcategory`
- "product" -> `product_name`
- "channel" -> `sales_channel`
- "region" -> `customer_region`
- "segment" -> `customer_segment`

Interpretation precedence:
- If a natural-language phrase can be mapped through the rules above, use the mapped canonical field instead of treating it as unsupported.
- If wording is shorthand or informal but the intent is clear, resolve it to the closest supported metric/dimension/question type instead of falling back to unsupported.
- If there is a direct conflict between older prompt behavior and these interpretation rules, these interpretation rules win.

Ranking detection:
- If the query includes "top", "bottom", "most", "least", "highest", or "lowest", treat it as ranking intent.
- If the query includes "best", "worst", "largest", or "smallest", also treat it as ranking intent when the question is clearly asking for ordered entities.
- When ranking is implied:
  - `questionType` MUST be `"ranking"`
  - `sort.by` MUST be `"metric"`
  - `sort.direction`: "top"/"most"/"highest"/"largest"/"best" -> `"desc"`; "bottom"/"least"/"lowest"/"smallest"/"worst" -> `"asc"`
  - Use the most specific applicable dimension from the wording:
    - "products" -> `product_name`
    - "customers" -> `customer_name`
    - "categories" -> `product_category`
    - "subcategories" -> `product_subcategory`
    - "channels" -> `sales_channel`
    - "regions" -> `customer_region`
    - "segments" -> `customer_segment`
  - `limit` MUST be extracted if explicitly stated.
  - If ranking is clearly implied but no numeric limit is stated, infer `limit = 5`.

Default behaviors:
- If no time range is specified, use `last_30_days`.
- If grouping is implied and ranking is not implied, use `grouped_breakdown`.
- If ranking is implied, use `ranking`.

Canonical unsupported pattern — only use when:
1. The request is not an analytics query
2. The request asks for free-form SQL, schema introspection, raw export, or unavailable data outside the supported surface
3. The request is predictive, forecasting, causal/explanatory, or requires external data
4. The required metric truly does not exist and cannot be mapped through the interpretation rules above
5. A required filter field or filter value is not in the supported surface
6. The time behavior is genuinely unresolvable

Do NOT use the unsupported pattern when:
- A synonym or alias can be mapped to a valid metric/dimension
- The query is a standard aggregation, breakdown, ranking, or time series over supported data
- The wording is informal but the intent is clear
- Minor ambiguity can be reasonably resolved by the interpretation rules and defaults above

Question type rules:
- `aggregate` requires `dimension = null`, `timeGrain = null`, `limit = null`.
- `grouped_breakdown` requires exactly one non-null allowed `dimension`, `timeGrain = null`, `limit = null`.
- `ranking` requires exactly one non-null allowed `dimension`, `timeGrain = null`, and `limit` from 1 to 50.
- `time_series` requires `dimension = null`, `limit = null`, and a valid allowed `timeGrain`.

Filter rules:
- Use at most 8 filters.
- Use at most one filter per field.
- Operators must come only from the supported operator set.
- `eq` requires exactly 1 value.
- `in` requires 1 to 20 values.
- Do not emit empty strings, blank strings, or empty value arrays.
- If the user asks for a filter field or filter value that is not in the supported surface, use the canonical unsupported pattern.

Time range rules:
- `timeRange.preset` must come from the supported preset allow-list.
- For non-`custom_range` presets, set `startDate = null` and `endDate = null`.
- For `custom_range`, both dates are required and must use `YYYY-MM-DD`.
- If the user specifies a specific calendar year (e.g. "in 2024", "for 2024", "during 2023"), use `custom_range` with `startDate = "YYYY-01-01"` and `endDate = "YYYY-12-31"`.
- If the user does not specify a time range, use `last_30_days`.
- If the requested time preset or grain is unsupported or unclear and cannot be reasonably resolved, use the canonical unsupported pattern.

Sort rules:
- `sort.by` must be `metric` or `dimension`.
- `sort.direction` must be `asc` or `desc`.
- For `aggregate`, emit `{"by":"metric","direction":"desc"}`.
- For `grouped_breakdown`, default to `{"by":"metric","direction":"desc"}` unless the user clearly requests alphabetical or dimension ordering.
- For `ranking`, use `sort.by = "metric"` only.
- For `ranking`, default to `{"by":"metric","direction":"desc"}` unless the user clearly asks for bottom/lowest/least ordering.
- For `time_series`, emit `{"by":"metric","direction":"desc"}`. Downstream compilation handles chronological ordering separately.
